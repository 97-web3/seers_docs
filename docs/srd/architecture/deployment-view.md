# 部署视图

## 1. 部署假设（与技术无关）
- 用户通过**客户端应用程序**（Web和/或移动端）访问系统。
- 客户端通过 HTTPS 与**公共 API** 通信。
- 核心功能可以实现为独立服务或模块化单体应用；接口契约保持稳定。

## 2. 部署图（逻辑）
```mermaid
flowchart LR
  subgraph 客户端
    CA[客户端应用程序]
  end

  subgraph 边缘
    API[公共 API 端点]
  end

  subgraph 核心
    S1[事件与市场能力]
    S2[交易能力]
    S3[投资组合能力]
    S4[配置能力]
    S5[结算/裁决能力]
    S6[媒体能力]
  end

  subgraph 外部
    SDP[体育数据提供商]
    WS[钱包/支付]
    OBJ[对象存储]
    ACP[管理/运营配置]
  end

  CA --> API
  API --> S1
  API --> S2
  API --> S3
  API --> S4
  API --> S5
  API --> S6

  S1 --> SDP
  S2 --> WS
  S3 --> WS
  S4 --> ACP
  S6 --> OBJ
```

## 3. 可用性预期（高层次）
- API 在体育赛事高峰期间应保持可用；错误处理必须能优雅降级。
- 状态变更操作（下注、创建市场）需要比只读端点更强的保证。
